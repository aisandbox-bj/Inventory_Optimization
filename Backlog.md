## Shipped 2026-06-28 (P1-rate timezone fix + HCE-table removal) — NOT yet pushed · pending operator validation
- **APP-FIX-P1-RATE** — the P1 consumption rate was **~17–20% low**. Root cause: `p1Months = monthsBetween(p1Start,p1End)+1`, and `monthsBetween` used **local** date getters on UTC-midnight-parsed dates, so in a west-of-UTC timezone `new Date('2025-04-01').getMonth()` reads as **March** → the default Apr 1→Aug 31 window counted **6** months, not 5 → P1 divided by one month too many. Same family as `APP-FIX-BACKCALC-TZ` / `APP-FIX-CHART-TZ`. Surfaced by the operator overlaying the dynamic period (APP-TREND-DYN) on P1 and seeing >20% gaps, "more pronounced over P1." **Fix (operator decision: fix monthsBetween + snap):** (1) `monthsBetween` → **UTC** getters; (2) the P1 window is **snapped to whole calendar months** in the pipeline *and* at intake (`snapMonthStart`/`snapMonthEnd`). **P2 unchanged** (its window is built as exactly `p2Months`, never uses `monthsBetween` — which is why P1 was the pronounced one). **Blast radius (deterministic-math change — surfaced):** every whole-month-window P1 rate rises exactly **20%** (÷6→÷5) to its correct value; on LV (69/82 with a P1 rate) that re-trips **3 rate-drop + 6 rate-rise** flags (corrections off the previously-buggy rate) and any traffic-light keyed off them. Verified live: 1001220 26.0→**31.2**, matches the dynamic overlay + an independent recompute; zero console errors. Files: `shared/pipeline.js`, `intake/intake.js`. No SCHEMA_VERSION change.
  - **Latent follow-up — ✅ FIXED 2026-06-29 (APP-FIX-ADDMONTHS-TZ):** `addMonths` (builds the P2 window) mixed local `get/setMonth` with UTC `toISOString` — same timezone-trap family, shifted the P2 edge by up to a few days for month-boundary run dates. Now UTC. See the 2026-06-29 section at the top.
