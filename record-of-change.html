Version management · rollback-safe

Record of Change

Every release is tagged on the GitHub repo and listed here. Each entry names what changed, what schema/file boundaries it touched, and the exact git command to roll back to any prior version. Releases marked dev are in-flight; they become released when the tag lands on main.
How to roll back

Reset to any tagged version

From a clone of the repo (e.g. /tmp/push-Inventory_Optimization), check out any tag to see the working tree at that version, then either copy the files back into your local App/v1/ directory or reset main to that tag. The user's working folder is not a git repo — rollback always happens through a clone, never via git init in the working dir. 

# Inspect a prior version (read-only checkout)
git clone https://github.com/aisandbox-bj/Inventory_Optimization.git /tmp/peek
git -C /tmp/peek checkout v1.0.0
# → now /tmp/peek holds the working tree at v1.0.0; copy what you need back

# Or hard-reset main to a tag (irreversible — only when intended)
git -C /tmp/peek checkout main
git -C /tmp/peek reset --hard v1.0.0
git -C /tmp/peek push --force-with-lease origin main

Local App/v1/ reflects whatever Claude has just written — it tracks the dev tip until the next release tag. Save a snapshot before destructive edits if uncertain.

v2.0.1 2026-05-12 released

TWEAK · Configurable Working Redundant thresholds

  • The PURPLE / Working Redundant classifier is now driven by three operator-configurable parameters instead of hardcoded literals:
    • wrSoftMonths (default 6) — stock-runway threshold for Possible WR (PURPLE soft).
    • wrHardMonths (default 9, lowered from previous hardcoded 12) — stock-runway threshold for Likely WR (PURPLE hard — write-down review).
    • wrMrpTypes (default ['PD']) — MRP types that trigger the WR check at all. Comma-separated list in the parameter editor; uppercase-normalised on save.
  • Defaults reflect operator input that 12 months was too lenient on PD items — 9 months of stock at P2 rate on a PD item is already a strong write-down candidate.
  • Action-text now interpolates the active threshold and the MRP type that tripped — e.g. "Likely Working Redundant — 11.3 months of stock at P2 rate (>9mo with PD). Review for destocking / write-down." Widening wrMrpTypes to include e.g. ZE produces an unambiguous message naming the specific code.
  • Editable in Settings → Parameter defaults (persistent) and per-intake in Step 5 (one-off override with the same dirty/reset UX as every other parameter). A new type:'list' input pattern was added to both parameter editors to support comma-separated MRP-type lists; first list-typed parameter in the app, generic and re-usable for future list parameters.
  • Fully back-compatible: older intake JSONs without the new keys fall through to factory defaults at pipeline time (guarded reads in assess() and validateParameters). No SCHEMA_VERSION change — still 1.0.0. Older intakes saved before v2.0.1 continue to load and run identically, picking up the new 9-month "likely" threshold on re-run (which may shift some materials from "possible" to "likely" WR — expected behaviour reflecting the tighter operator standard).
  • User-manual §7f updated to reflect parameter-driven phrasing (rules 4 & 5 in the traffic-light table + explanatory paragraph).
Files touched: shared/canonical-schema.js (APP_VERSION → 2.0.1, +3 defaults, +3 descriptions, +4 guarded validators) · shared/pipeline.js (assess() signature gains params, WR block now configurable, call site passes params) · settings/settings.js (3 field defs + new list input type) · intake/intake.js (same 3 field defs + same list input + array-aware dirty detection) · user-manual.html §7f · this file.
Rollback: revert pipeline.js WR block to literals 6 and 12, drop the three keys from FACTORY_DEFAULTS. Older intakes continue to work either way. Or simply git checkout v2.0.0 if that tag is later added retroactively. The pre-v2.0.1 HEAD on origin/main is the rollback point.
v2.0.0 2026-05-12 dev

Reuse-modal redesign + mandatory Assessment Name (2026-05-12 pm)

  • Reuse-data tile relocated from above the metadata header to below the Intake Engine header. Reads better — name your assessment first, then decide whether to reuse data.
  • Inline picker → modal. Tile now shows a compact "↻ Pick source + datasets…" button that opens a proper bubble.
  • Two-step modal:
    • Step 1 — Pick source intake: dropdown of all saved intakes
    • Step 2 — Pick datasets: checkbox per dataset (MB51 / IW39 / Fleet Master / Inventory Master / userList / materialVendor / leadTimes). Each row shows the row count from the source intake. Datasets not present in the source are greyed out and disabled.
  • Smart defaults: heavy common files (MB51, IW39, Fleet, Inv Master) are pre-checked. Specifier files (userList, materialVendor, leadTimes) are un-checked by default — the operator picks what they actually want for the new batch.
  • Assessment type is NO LONGER auto-inherited from the source intake. The operator picks a fresh assessment type in Step 0 after loading. Fixes the previous behaviour where reusing a userList-based intake force-applied the userList type.
  • Mandatory Assessment Name: the name field is now required (red asterisk + required/aria-required). Validates on Save, Download, and Open-in-Analysis. Empty submission triggers an inline error message, focuses + scrolls to the field, and toasts a red warning. Typing into the field clears the error live.
  • Auto-suggested batch name ({source} — batch HHMM) only fills when the field is empty — never overwrites a name the user typed.

Locale fix — host-clock time + CAD currency

  • Time was off by your UTC offset. Every new Date().toISOString() through the app converted local clock → UTC, which is hours later than your wall clock west of UTC. After ~5pm local time, the "Run date" was already showing tomorrow's UTC date. Same for "Created at", "Exported at", PDF footer, mass-review timestamps.
  • New shared/locale.js module with AppLocale.localDateISO(), localDateTimeISO(), localStampCompact(), localDateTimeLabel() — all read from the host machine's clock (Date object's local getters), never converting to UTC. Loaded on every page.
  • Replaced 20+ usages of toISOString() across analysis.js, intake.js, shared/canonical-schema.js, shared/excel.js, shared/mass-llm.js with the appropriate local helper.
  • Parser kept on ISO — the parser's toIsoDate() still uses toISOString().slice(0,10) for Excel date-serial conversion (no timezone ambiguity for stored dates from SAP). Only the app-generated timestamps switched.
  • CAD currency formatting via AppLocale.fmtCAD() (Intl.NumberFormat 'en-CA', no fractional digits by default). Applied to:
    • Parameter Search preview tile → "Total on-hand value (CAD)"
    • Material detail panel → new "Stock value (CAD)" stat tile
    • Excel per-material sheet → new "Stock Value" row in the field/value/notes table
    • PDF footer → "CAD" tag alongside the timestamp + version
  • Helper also exposes EXCEL_CURRENCY_FORMAT = "$"#,##0 for future numFmt usage on Excel cells.

Batch-mode workflow — reuse common data + manual paste for user lists

  • New "Reuse common data" panel at the top of the Intake page (only shown when at least one intake is already saved). Operator picks a previously-saved intake from a dropdown → MB51 / IW39 / Fleet / Inventory Master / Material→Vendor / Lead Times slots are all hydrated from that intake's canonical JSON. Upload zones show ↻ reused from "{name}" instead of "drag here". DQ gate auto-re-runs.
  • Each subsequent batch only needs the operator to: pick a new assessment type + new scope + (optionally) tweak parameters + save as a new named intake. Heavy data stays in memory across batches — no re-uploading.
  • Default new-intake name auto-suggested as {source name} — batch HHmm; operator can rename. Parameters carry over as a starting point (per-run override mechanic unchanged). Scope is intentionally reset — that's the point of a new batch.
  • User-list file is now OPTIONAL for User-list assessment type. The drop zone is still offered (badged "(optional)") but isn't required for DQ pass. Required-sources for userList type is now just ['mb51', 'inventoryMaster'].
  • Direct paste in Step 4's Manual pane — operator types / pastes materials straight into the textarea once MB51 + Inventory Master are loaded. The Manual pane now shows an inline hint explaining "paste materials directly here — no file upload needed". If both a file AND a paste are present, the paste wins (most recent edit is source of truth).
  • Grey-out logic switched to data-needed-by — the Step-0 grey-out now reads each drop's HTML attribute (which can list more types than the hard-required set), so the userList drop remains active-but-optional under the userList type instead of being greyed.
  • buildJson() stops force-overwriting scope.manual.materials from the userList file; it now only auto-derives if the manual textarea is empty AND a userList file is present (defensive fallback).

PDF Pack export + Right-click mark workflow

  • Right-click any row in the analysed material list to open a custom context menu with four actions: Mark for LLM review, Unmark from LLM review, Mark for PDF print, Unmark from PDF print, plus Open detail panel. The menu auto-swaps "Mark"/"Unmark" based on current state.
  • Visual indicators on marked rows: small magenta ✦ badge for review-marked, amber ⤓ badge for PDF-marked, plus a coloured stripe on the left edge of the first cell.
  • Mass LLM Review modal now pre-checks materials marked for review (capped at the 50-material limit).
  • New "Export PDF Pack" button in the Bulk operations row. Opens a selection modal pre-checked from PDF-marked materials. Three quick-selects: Select all visible, Select marked only, Clear. Counter updates live.
  • PDF generation — A4 portrait, one page per material. Each page contains: navy header band (assessment / bucket / run date / page X of N), large material number + description, traffic-light pill, algorithmic recommendation banner, embedded consumption chart (1100×430 px), two-column stat grid (P1/P2 rates / adj P2 / rate change / total / pattern / stock / runway / WO count / multi-model), MRP Settings Comparison table with amber highlighting on rows where recommended differs from current, HCE events table (orange-tinted), Inv Adj events table (purple-tinted), and a footer with timestamp + repo URL.
  • Buttons show counts when materials are marked — e.g. "✦ Mass LLM Review · 12 marked" / "⤓ Export PDF Pack · 7 marked" so the marked sets are visible at-a-glance.
  • Built on jsPDF 2.5.1 + jsPDF-autoTable 3.8.2 (CDN). No html2canvas dependency — the chart embeds via the existing AppChart.toPng() path, all other content uses jsPDF's native text + autoTable for crisp, vector-friendly output.

Dashboard — "Clear session data" button

  • New button at the bottom of the Dashboard (under Recent intakes) with its own framed card and amber-warn styling: ⌫ Clear session data.
  • Wipes every intake.* and intakes.* key in storage — saved intakes, the current-intake slot, the intakes index — while preserving every settings.* key (parameter defaults, LLM provider keys, prompt template, column-alias overrides).
  • Distinct from the Settings page's "Restore factory" (which is the inverse — wipes settings, keeps intakes) and "Wipe all local data" (which destroys both).
  • Two-step confirm: spells out exactly which keys will be removed and which preserved before proceeding. Reloads the page on completion so the Recent intakes table refreshes.
  • New helper AppConfig.clearSessionData() returns the number of keys removed for the confirmation toast.

Inventory Adjustment detection + exclusion

  • New detector in shared/pipeline.js: groups MB51 issue rows by posting date, computes mean μ and stddev σ of daily counts, flags dates with count ≥ μ + N·σ as candidates. N is configurable via parameters.invAdjSigmaThreshold (default 5).
  • Auto-popup modal after pipeline runs when candidates exist + no prior confirmations. Shows: days analysed, avg daily issues, stddev, threshold. Candidate table with date / day-of-week / count / inline avg-vs-day bar viz / σ above mean / materials affected / total qty. Operator ticks dates to exclude. Confirming saves to parameters.invAdjConfirmedDates and re-runs the pipeline.
  • Pipeline rate exclusion: calcPeriodRate() accepts an optional excludeDates Set. P1 and P2 rates skip rows on confirmed Inv-Adj dates. Adjusted P2 (HCE-excluded) ALSO excludes Inv-Adj dates — both signals stack.
  • Per-material output: new invAdj array on each result, shaped like the existing hceP2 array (date, qty, order(s), reason). Rendered as a purple-tinted table on the material detail panel and a dedicated section on the Excel per-material sheet.
  • Chart annotation: vertical dashed purple line at each Inv-Adj date with "INV ADJ" label — distinct from amber HCE markers.
  • Settings + Intake §5: invAdjSigmaThreshold added to the parameter editor with full descriptor.
  • "Inv Adj review" button on the analysis page (Bulk operations row) re-opens the modal anytime.
  • Confirmations persist on the canonical JSON — reloading the intake remembers the operator's decisions.

v1.1.0 2026-05-12 released

Pipeline — PURPLE / Working Redundant classification + few-events overlay

  • New PURPLE traffic light for Working Redundant (WR) detection on PD materials:
    • PD + stock runway > 6 months at P2 rate → Possible WR (review stocking necessity).
    • PD + stock runway > 12 months at P2 rate → Likely WR (review for destocking / write-down).
    • V1 materials never trip purple — V1 is consumption-driven so over-stock self-corrects.
  • Few-events overlay — after the primary classification, if the material had ≤ 2 unique issuing work orders in the analysis window AND the result was GREEN or BLUE, it's upgraded to ORANGE with action "Only N event(s) — pattern unreliable, manual review". RED / PURPLE / GREY are not overlaid (they're already review-priority). Rationale: 1–2 issues can produce a statistically "fine"-looking rate by luck, not by pattern.
  • Each material result now carries woCount (unique issuing work orders) and fewEvents (boolean).
  • Decision tree expanded from 8 → 10 rules; full table re-documented in the User Manual §7f.
  • PURPLE wired through: pipeline summary counters, analysis-page TL tile + filter button + dot + pill, Excel cell fill (#9B59B6, white text), Summary sheet ordering (Green, Blue, Orange, Red, Purple, Grey), cross-bucket Summary in combined workbook.

User Manual + chart sparse-data fix

  • New user-manual.html — deep operator manual (TOC + 13 sections) covering quick-start, assessment types, data files / SAP movement-type semantics, scope strategies, Parameter Search, full analysis methodology (net consumption, P1/P2 windows, HCE rules, lumpy/smooth, recommended Min/Max, the 8-rule traffic-light tree, bucketing + MULTI detection), chart anatomy + why charts may not match expectations, Excel deliverable structure, LLM review, troubleshooting, and versioning / rollback.
  • Chart bug fixed — earlier versions rendered "no consumption data" when a material's cumulative series collapsed to a single posting date, even though P1/P2 rates and recommendations existed. The cumulative groups by unique date; if all transactions hit one calendar day, only one point existed and the chart bailed. Now the cumulative is padded with a synthetic {p1Start, 0} anchor when needed (and a trailing flat plateau to p2End) so the step line + trend chord always render. The "no consumption" message now reads "no consumption transactions found" and only triggers on a true zero-transactions case.
  • Manual linked from every page's nav.

Analysis — full-width material table + Excel-style sortable / filterable headers

  • Material list now spans the full screen width; detail panel sits below instead of beside, so all eight columns are visible without horizontal scrolling.
  • Every column header has a sort label (toggles asc/desc) AND a filter caret ▾ that opens a popover. Three popover styles match column type:
    • Set picker (TL, Material, MRP, Pattern) — checkbox list of every distinct value, with Select all / Clear / Apply.
    • Range picker (Total, P1/mo, P2/mo, Rec Min, Rec Max) — numeric min / max inputs.
    • Text picker (Description) — contains-substring search.
  • Filters AND together; active filter caret highlights in the brand accent so it's obvious which columns are filtered. Bucket switch clears filters automatically.
  • Footer line below the table reads "X of Y materials shown".

Excel deliverable — reference-matched per-bucket workbook

  • Per-bucket workbook now mirrors the reference 775G_Analysis.xlsx deliverable exactly:
    • Index sheet — title row merged across A1:Q1, 17-column header on row 2 (Material No., Description, Total Consumed, P1/P2 Rate, Rate Change %, Stock, MRP Type, Current/Recommended Min/Max, Traffic Light, Action Required, HCE Flag (P2), Adj P2 Rate, Multi-Model Flag), frozen at row 2, AutoFilter on the header row (Excel-native filter dropdowns).
    • Hyperlinks — Material No. cell on Index → that material's sheet; "← Index" back-link on every per-material sheet.

Field / Value / Notes table with 21 rows (Material No., Description, Material Group, Manufacturer, Multi-Model Flag, Total Consumed, P1 Net Qty + Rate, P2 Net Qty + Rate, Adj P2 Rate, Rate Change, Current Stock, Runway @ P2, MRP Type, Current / Rec Min + Max, Traffic Light, Action Required).     • Chart image embedded at the F2 anchor (~1100 × 470 px), matching reference width.
