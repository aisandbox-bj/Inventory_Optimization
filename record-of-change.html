Version management · rollback-safe

Record of Change

Every release is tagged on the GitHub repo and listed here. Each entry names what changed, what schema/file boundaries it touched, and the exact git command to roll back to any prior version. Releases marked dev are in-flight; they become released when the tag lands on main.
How to roll back

Reset to any tagged version

From a clone of the repo (e.g. /tmp/push-Inventory_Optimization), check out any tag to see the working tree at that version, then either copy the files back into your local App/v1/ directory or reset main to that tag. The user's working folder is not a git repo — rollback always happens through a clone, never via git init in the working dir. 

# Inspect a prior version (read-only checkout)
git clone https://github.com/aisandbox-bj/Inventory_Optimization.git /tmp/peek
git -C /tmp/peek checkout v1.0.0
# → now /tmp/peek holds the working tree at v1.0.0; copy what you need back

# Or hard-reset main to a tag (irreversible — only when intended)
git -C /tmp/peek checkout main
git -C /tmp/peek reset --hard v1.0.0
git -C /tmp/peek push --force-with-lease origin main

Local App/v1/ reflects whatever Claude has just written — it tracks the dev tip until the next release tag. Save a snapshot before destructive edits if uncertain.

APP-E1 · Stockout-aware consumption drop detection

  • Why it shipped. Operator finding 2026-05-15: "you cannot consume what you don't have." Many cases of uncharacteristic zero consumption — especially in P2 — are stock-out artefacts, not real demand drops. Previously the pipeline's rateDropFlag treated any P2 ≤ 0.6 × P1 as a sharp drop without distinguishing the cause. Lowering Min/Max on a stockout-driven "drop" makes the stockout self-fulfilling. Fix surfaces the cause + the visual signal needed to verify it.
  • New parameter socBackCalcMonths (default 6). The SOH back-calc window in months, anchored at each material's last consumption event and extending backward, then forward to today. Operator-overridable on intake Step 5 and in Settings → Parameter defaults. Floor constraint: must be ≥ p2Months (so the diagnostic window always covers the rate-change comparison period).
  • Anchor design. The window anchors at lastConsumptionDate = max(postingDate where mvt ∈ {261, 201}) per material — NOT at today. For healthy materials with continuous consumption, last-consumption ≈ today, so the window is naturally [today − X, today]. For materials with long zero-consumption tails, the window anchors at the date consumption stopped — exactly the run-up where the diagnosis lives. Operator's framing: "did I have a snap consumption increase, or did my replenishment process fail?" The visual answers the question.
  • New shared module shared/inventory-back-calc.js (pure, deterministic, no side effects). Walks MB51 backwards from current SOH applying +109 / -261 / -201 / +101 / -102 / +262 / +202 rules (all other movement types deliberately ignored — adjustments aren't real demand or supply). Designed to be called from Trace's D14 inventory back-calc overlay too — math primitive lives once, both tools call it.
  • Pipeline output per material now carries five new fields: lastConsumptionDate, stockOnHandSeries (daily SOH back-calc), stockoutWindows (periods where SOH ≤ 0, with day-counts), rateDropCause (STOCKOUT_DRIVEN / GENUINE_DEMAND_DROP / null), stockoutDrivenDrop (boolean). Augments — doesn't replace — the existing rateDropFlag: the drop is still flagged, but the cause is named so downstream consumers (operator, LLM) narrate it correctly.
  • Consumption chart visualisation (the operator's key diagnostic): on the analysis page material detail, the existing cumulative chart gains three overlays. (a) Red wash bands for stockout windows (SOH ≤ 0), with crisp edge lines and a "STOCKOUT · Nd" label. (b) Violet SOH back-calc line on a SECONDARY right-side Y-axis (SOH is units-in-stock, not cumulative consumption — distinct scale). (c) Orange dashed vertical "LAST CONSUMPTION" marker at lastConsumptionDate. Legend extends with SOH + STOCKOUT entries when relevant.
  • Material detail panel adds three new stat cells when applicable: Last consumption (date) · Stockouts in back-calc window (count + total days, green if none / red if any) · Drop cause (Stockout-driven in red / Genuine demand drop in orange). Operator sees the diagnostic at-a-glance without running an LLM review.
  • LLM prompt context extends the per-material STATISTICS block with lastConsumptionDate / stockoutSummary / rateDropCause. The rateChangeFlag token now reads "⚠ SHARP DROP — STOCKOUT-DRIVEN (consumption stopped because stock ran out, not because demand dropped)" or "⚠ SHARP DROP — GENUINE DEMAND DROP" when the cause is known. Chart description updates accordingly. New top-priority WATCH FOR bullet: "STOCKOUT-DRIVEN DROP — do NOT recommend lowering Min/Max — the material still has demand; the supply chain failed." New signal name stockoutDriven in the signals[] array.
  • No SCHEMA_VERSION bump. All additions are additive to per-material output; the canonical intake JSON shape is unchanged. APP_VERSION → 2.1.3-dev.

FIX-1 · Wrong movement-type set on the stock back-calc + jargon copy

  • Bug 1 (math — caught on first live look): the per-material transactions handed to the back-calc were pre-filtered at the bucketing step to consumption movements only (261 / 201 / 262 / 202). Goods receipts (109) were never visible to the back-calc, so walking backward through only-issues made the stock line climb monotonically as you looked further back — the "infinite supply" artefact operator flagged. Fix: pre-group the FULL json.data.mb51 by material once in runPipeline, and pass each material's raw MB51 slice to backCalcSOH. Receipts and issues both reach the back-calc; the line now oscillates as it should.
  • Bug 2 (movement-type semantics — operator-corrected): the original MVT_SIGN map included 101 / 102 as receipts. These are 3PL events, not site events — they land material at the third-party holding location, not into the site warehouse that Inventory Master's totQtyOh tracks. Including them would falsely show site stock spiking up whenever a vendor delivered to the 3PL. Fix: dropped 101 / 102 from MVT_SIGN. The map is now {109:+1, 261:-1, 201:-1, 262:+1, 202:+1} — only movements that affect site unrestricted stock. The module-header comment explains the site-vs-3PL distinction and notes that operator-specific transfer types (311, 411, etc.) can be appended if a particular site uses them as supply paths onto site.
  • Bug 3 (sloppy copy): the new parameter was labelled "SOH back-calc months" and its help text contained the internal IDs "APP-E1" and the acronym "SOH" — operator-facing jargon. Fix: label renamed to "Stock history window (months)". Helper text rewritten in plain English explaining what the window covers, what default value (6) it carries, the floor constraint (≥ P2 months), and what stockout-driven vs genuine-demand-drop actually means. Chart legend label changed SOH → STOCK; right-axis label changed SOH (UNITS) → STOCK ON HAND. Analysis-page stat tile renamed Stockouts (back-calc) → Stockouts in window. LLM prompt template's chart description rewritten to call the violet line "stock on hand reconstructed from MB51 movements (SITE stock only, 3PL receipts excluded)".
  • No schema change. No new files. APP_VERSION stays at 2.1.3-dev (this is a same-day correction, not a new version).

APP-E11 · P2 stockout-anchored math + chart polish (APP-E1 follow-up)

  • Why it shipped. Operator's live review of APP-E1 surfaced misleading P2 verdicts on materials that had been in an ongoing stockout. Negative stock-on-hand on the chart was the visual signature of the rate comparison being misled — when the recent-rate window is dominated by zero-consumption stockout days, "demand dropped" is the wrong verdict; the real issue is supply continuity. Also: the chart was visually cramped after APP-E1 layered three overlays onto it, and the SOH line had no caveat clarifying it is back-calculated rather than pulled from SAP.
  • (a) Math correctness — P2 window anchoring when stockouts dominate. The pipeline now examines each material's stockout history BEFORE computing the P2 rate. If the back-calc tail is in an ongoing stockout of more than 7 days, the P2 window re-anchors at lastConsumptionDate (step back p2Months from there), instead of the default anchor at runDate. The recent-rate window then represents the operator's last observation of true demand before the stockout began — not the zero-consumption stockout days themselves.
  • STOCKOUT-DOMINATED classification. If MORE THAN ONE stockout window falls inside the chosen P2 window, the rate verdict is judged untrustworthy. A new traffic-light path forces GREY · "Stockout-dominated recent-rate window — multiple stockouts inside the P2 window make the rate unreliable. Verify supply continuity before changing Min/Max." When STOCKOUT-DOMINATED, rateDropFlag / rateRiseFlag / rateChange are suppressed and the LLM context includes a top-priority WATCH-FOR token: "⚠ STOCKOUT-DOMINATED RECENT-RATE WINDOW — multiple stockouts inside P2 make the rate untrustworthy; verify supply continuity, do NOT recommend Min/Max change on the basis of P2 rate."
  • New helpers in shared/inventory-back-calc.js: isTailInOngoingStockout(windows, runDate, minDays=7) · countStockoutsInRange(windows, rangeStart, rangeEnd) · chooseP2Window({runDate, lastConsDate, p2Months, tailInOngoingStockout}). All pure, deterministic, no side effects.
  • Per-material output adds three fields: p2AnchorMode ('runDate' | 'lastConsumption'), p2StockoutCount (integer), stockoutDominated (boolean). The bucket-level p2Start / p2End still reflect the default (runDate-anchored) window; per-material p2Start / p2End reflect the actual chosen window.
  • (b) Consumption chart — 30% wider. The analysis-page chart canvas widens from 720 → 936 SVG units (height unchanged at 320). Same parent container, but the overlays now breathe.
  • (c) Legend with line toggles. Two checkboxes above the chart — Consumption and Stock on Hand. Each toggles a class on the chart-host (.hide-cum / .hide-soh); CSS rules hide the tagged SVG layers (.chart-cumulative-line and the <g class="chart-grp-soh"> wrapper added in chart.js). Toggle state re-applies on every render so switching materials preserves the operator's view.
  • (d) Back-calc caveat caption. Italic note below the chart: "Stock-on-hand line is back-calculated from MB51 movements (site stock only, 3PL receipts excluded) — not pulled from SAP." Prevents over-trusting the violet line as ground-truth stock data.
  • No SCHEMA_VERSION bump. All additions are additive. APP_VERSION stays at 2.1.3-dev (APP-E1 follow-up — not a new version).

APP-E16 · Production back-calc — broaden MVT scope per CoWork audit (site-specific) + row-level sign for 309/411

  • Why it shipped. The production back-calc in shared/inventory-back-calc.js was using the original 5-MVT baseline (109/261/201/262/202) — the same narrow scope that produced 175 calculated-negative SOH values and ~7% material-variance rate in the SOH audit tool. CoWork's 2026-05-16 review identified the missing scope and the row-level sign issue for 309/310/411/412. Operator verified the fix in the audit tool on a 57-material filtered sample: calculated<0 went from 1 → 0, |variance|>0 from 15 → 10, and the 1003195/1014958 phase-out pair reconciled from −218/+137 down to ~0/+4. This entry ports the verified audit-tool fix into production Tune so every analysis run from this version onward uses the corrected math.
  • Site override — 101 / 107 / 109. At this client, 101 is the 3PL (off-site) receipt — NOT a site receipt despite standard SAP semantics. 107 is the in-transit leg from 3PL toward site (also doesn't touch site WH yet). Only 109 is the actual site WH receipt. Comment in MVT_SIGN's header explicitly says "do NOT 'fix' back to standard semantics" so a future maintainer reading SAP reference materials doesn't undo the override.
  • New MVT_SIGN — 18 entries. Consumption (261/201/221/291/551/543) + reversals (262/202/222/292/552); site receipts (109/712); receipt reversals (110); count variance (711); subcontracting (541/542). Verified site-specific scope per CoWork review.
  • New DIRECTIONAL_MVTS set. 309 / 310 material-to-material and 411 / 412 plant-to-plant transfers — SAP emits source row as negative qty, destination row as positive qty under the SAME MVT. A fixed sign in MVT_SIGN would mis-sign one leg. For these, backCalcSOH() trusts the row's qty sign as-is. THIS is what fixes the phase-out / replacement-material reconciliation pattern.
  • New DELIBERATELY_EXCLUDED set. 3PL flow (101/102/107/108/641/642), initial stock setup (561/562), bin/storage-location moves (311/312/313), QI/block reclassifications (321/322/343/344), non-unrestricted scrap/variance (553-556/703-708/713-718), sales/outbound/samples (601/602/653/331/332). Documented for future-audit transparency — distinguishes "considered and ruled out" from "never reviewed."
  • Three-path signed-delta logic in backCalcSOH(): (a) DIRECTIONAL MVT → row's qty sign as-is; (b) MVT in MVT_SIGN → fixed sign × |qty|; (c) otherwise (excluded or unknown) → row skipped. Same logic as the audit tool, byte-identical math.
  • LLM prompt chart description (config.js FACTORY_PROMPT_TEMPLATE) updated to describe the broader back-calc scope: "site WH unrestricted only; back-calc includes site receipts/issues, scrap/write-off, count variance, project issues, and subcontracting movements; 3PL receipts and bin/storage-location moves within plant are excluded; material-to-material and plant-to-plant transfers are read with their row-level sign." Prevents the LLM from misinterpreting a clean back-calc as missing data.
  • Public API additions. InventoryBackCalc now exposes DIRECTIONAL_MVTS and DELIBERATELY_EXCLUDED alongside the existing MVT_SIGN so any caller (Trace's planned D14 back-calc overlay, future audit tooling, operator scripts) can introspect the scope.
  • What this DOES NOT change. pipeline.js ISSUE_TYPES / RETURN_TYPES ({261,201} / {262,202}) stay UNCHANGED — they drive the P1/P2 consumption-rate calc, which is a separate concern from the back-calc. Broadening those would shift Min/Max recommendations across every material with 221/291 consumption; out of scope for this chunk pending separate operator + CoWork review. The back-calc and the rate calc now use different MVT scopes; that is intentional and consistent with how the audit tool was reviewed.
