Version management · rollback-safe

Record of Change

Every release is tagged on the GitHub repo and listed here. Each entry names what changed, what schema/file boundaries it touched, and the exact git command to roll back to any prior version. Releases marked dev are in-flight; they become released when the tag lands on main.
How to roll back

Reset to any tagged version

From a clone of the repo (e.g. /tmp/push-Inventory_Optimization), check out any tag to see the working tree at that version, then either copy the files back into your local App/v1/ directory or reset main to that tag. The user's working folder is not a git repo — rollback always happens through a clone, never via git init in the working dir. 

# Inspect a prior version (read-only checkout)
git clone https://github.com/aisandbox-bj/Inventory_Optimization.git /tmp/peek
git -C /tmp/peek checkout v1.0.0
# → now /tmp/peek holds the working tree at v1.0.0; copy what you need back

# Or hard-reset main to a tag (irreversible — only when intended)
git -C /tmp/peek checkout main
git -C /tmp/peek reset --hard v1.0.0
git -C /tmp/peek push --force-with-lease origin main

Local App/v1/ reflects whatever Claude has just written — it tracks the dev tip until the next release tag. Save a snapshot before destructive edits if uncertain.

APP-E21 · Intake → Analysis memory-bloat investigation (read-only — findings + reduction plan)

  • Method. Traced the handoff + analysis state with file:line evidence. No code changed this chunk (per scope — diagnosis first, reduction follows).
  • Primary culprit. The full raw canonical JSON — the entire data.mb51 (100k–300k+ rows), data.iw39, data.inventoryMaster — is written to the intake.current handoff slot (intake/intake.js) and held in analysis state.json (analysis/analysis.js) regardless of how few materials are in scope. A 6-month MB51 dominates the footprint (the operator's ~15 MB; larger for big extracts).
  • Important correction. This raw MB51 cannot simply be stripped from the handoff. Analysis re-runs the pipeline from raw (analysis.js → runPipeline(state.json) reads json.data.mb51), and Inventory-Adjustment detection computes its σ distribution across the full set of MB51 dates — filtering or blanking MB51 would change which dates flag as Inv-Adj and therefore change the recommendations. So the dominant cost is largely intrinsic to the deterministic re-run design.
  • Secondary culprits (safer to reduce). Per-material stockOnHandSeries (~180-point daily series ≈ 6.5 KB/material) + stockoutWindows retained for every qualifying material (~80 KB at 10 materials, ~800 KB at 100); plus a cumulative series stored on each material that analysis.js never reads.
  • Reduction plan (queued as APP-E21b), ranked by safety:
    1. Safe: drop the unused cumulative field from the per-material result (verify no refs in chart.js/excel.js first). Small, free.
    2. Safe-ish: MB51 column-prune at parse time — keep only fields the pipeline + chart + Trace consume; drop dead carry-through columns (~30–50% per-row shrink). Needs a field-usage audit.
    3. Medium: compute stockOnHandSeries lazily (only for selected/exported materials). The chart, LLM prompt, Excel and PDF all consume it, so it must be available when those run — needs a per-material MB51 slice + cache.
    4. Rejected as unsafe: blanking data.* in the handoff — breaks the analysis re-run + the Inv-Adj σ math.
    5. Architectural (defer): hand the computed result from intake → analysis instead of re-running from raw, or store raw MB51 once and reference it — removes the double-hold but is a larger change.
  • Status. Investigation only; no code changed. Reduction work queued as APP-E21b.

  • Operator finding 2026-05-25. The audit tool was running in the same timezone, so it shared the bug. Self-checking math against itself.

    • addDays() — unchanged. Adding 86 400 000 ms to a UTC midnight yields the next UTC midnight; no local-time interaction.
    • addMonths() — setMonth/getMonth → setUTCMonth/getUTCMonth.
