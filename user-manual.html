01Overview

Calibre Tune replaces the legacy Python-script + Excel-handoff workflow with a self-contained, browser-only tool. Drop SAP exports on the Intake page, the app validates them, you pick a scope, it runs the consumption-profiling pipeline, you review per-material findings (with an optional LLM second opinion), and it ships Excel + PDF packs matching the legacy deliverable shape.

The app is deterministic. Same inputs + same parameters always produce the same canonical JSON, the same recommendations, and the same Excel output. The LLM, when invoked, only annotates — it never changes the numbers.

Pages (all reachable from the nav bar):

  • Dashboard (index.html) — recent intakes, per-row delete, "Clear session data".
  • Intake (intake/intake.html) — upload → schema-map → DQ gate → scope → parameters → review → export.
  • Analysis (analysis/analysis.html) — runs the pipeline, surfaces per-material detail + chart, hosts the LLM review, exports Excel / PDF / JSON.
  • Trace (trace/trace.html) — Calibre Trace, the procurement-chain timeline, ported in as a sibling page (see §13). Reads PR History + MB51 from the same intake.
  • Settings (settings/settings.html) — parameter defaults, LLM keys, Operational Context, prompt template, alias overrides, multi-plant toggle, maintenance (see §12).

Shared engine modules live in shared/: pipeline.js (the deterministic analytical engine), inventory-back-calc.js (stock-on-hand reconstruction), chart.js, excel.js, llm.js / mass-llm.js, parsers.js, canonical-schema.js, client-context.js, config.js, storage.js, locale.js.

The canonical JSON is the contract Intake produces one canonical JSON (schema version 1.0.0). Analysis and Trace both consume it. The math deliverable carries no LLM metadata — verdicts, provider info, and client identifiers never touch the canonical JSON by design.

02Quick start

  1. From the Dashboard, click Intake.
  2. Assessment name (mandatory ★) — type a name in the header. Saving and "Open in Analysis" both refuse to proceed without one.
  3. Data-alignment gate — on a fresh assessment a modal asks you to confirm all uploads describe the same plant / date-range / material universe / fleet scope before any upload is accepted (see §3).
  4. Batch mode (optional) — if a previous intake already has the heavy data loaded, use ↻ Pick source + datasets… to hydrate MB51 / IW39 / Fleet / Inventory Master without re-uploading.
  5. Step 0 — Assessment type. UNIT/FLOC, User list, or Parameter Search. Non-applicable upload zones grey out.
  6. Step 1 — Upload. Drag SAP exports onto the highlighted drop zones. Each card shows row counts, a plant chip, and cross-file match percentages.
  7. Step 2 — Schema mapping. Confirm auto-mapped headers; override any unmatched columns.
  8. Step 3 — Data Quality gate. Green = continue, amber = warnings shown but continue, red = halt and fix.
  9. Step 4 — Scope. Fleet / Manual / By Classification / By Vendor (or Parameter Search in Step 4b). The Manual list accepts materials OR work orders with auto-detect + override.
  10. Step 5 — Parameters. Per-run override of the saved Settings defaults. Amber cells = overridden. Reset reverts without touching Settings.
  11. Step 6 — Final scope summary. Confirms exactly how many materials will be analysed, bucket-by-bucket, with mismatch advisories.
  12. Step 7 — Review & Export. Save to local storage, download .json, or "Open in Analysis Engine →".
  13. On Analysis: pick a bucket, filter/sort the material list, click a row for the detail panel. Right-click a row to mark for LLM review or PDF print. Export this bucket / Export ALL / PDF Pack / Mass LLM Review.
  14. On Trace (if you uploaded PR History): pick a material and read its procurement chain, phase distribution, funnel, and volume views (see §13).

Data-alignment acknowledgement gate (APP-E6)

On a fresh assessment, a "Data alignment check" modal fires before any upload is accepted. It is gated — only the "I confirm the uploads are aligned" button dismisses it. You confirm four things:

  • Plant / location / warehouse — every file filtered to the same plant set. Total-inventory + plant-only-consumption is the classic mismatch.
  • Date range — MB51 movement window and IW39 work-order window cover the same period.
  • Material universe — Inventory Master is a superset of every material in MB51 (missing master records cause silent drops at parse time).
  • Fleet / equipment scope — Fleet Master lists every unit appearing in IW39.

Manual list: materials OR work orders (APP-E22)

The Manual scope textarea accepts a paste of either material numbers or work-order numbers. Three buttons above it — Auto · Materials · Work Orders — control interpretation. Auto guesses from how the pasted values match Inventory Master / MB51 / IW39 and shows a confidence chip; you can override with one click. When the list is work orders, the scope resolves to the set of materials those orders touched in MB51.

04Data files & SAP exports

MB51 — Material Movements

The transactional backbone — goods issues, returns, and (for the stock-on-hand line) goods receipts.

  • postingDate — Posting Date / Pstng Date
  • order — Order (joins to IW39)
  • material — Material / Material Number
  • description — Material Description
  • quantity — Quantity (taken as absolute value; sign comes from movement type)
  • movementType — Movement Type / MvT
  • plant — Plant (new with the Fiori migration; used for the plant chip + cross-file consistency)
  • purchaseOrder — Purchase Order (used by Trace to join receipts to POs)
CodeMeaningSignUsed by
261Goods issue to order+ consumptionP1/P2 rates, cumulative, SOH back-calc
201Goods issue to cost center+ consumptionP1/P2 rates, cumulative, SOH back-calc
262Reversal of 261 (return)− returnP1/P2 rates, cumulative, SOH back-calc
202Reversal of 201 (return)− returnP1/P2 rates, cumulative, SOH back-calc
109Goods receipt (site)+ stock inSOH back-calc only (not a consumption rate). See §7c.
Other movement types are ignored for rates. The DQ gate warns if any unexpected types appear, and warns (not blocks) if 109 receipts are absent — the stock-on-hand line will then show drawdowns only.

The DQ gate halts if MB51 spans < 12 months or has no 261. It warns if 262 is missing (net = gross) or 109 is missing (see §7c).

IW39 — Work Orders

Bridges MB51 transactions to fleet equipment. Required only for UNIT/FLOC.

Equipment-unit roster. Required for UNIT/FLOC. model defines buckets; sortField bridges to IW39; unitType / manufacturer / functLocDescription are display-only.

05Scope selection

Step 4 drives which materials get bucketed and how. The available tabs depend on the assessment type.

Fleet

Pick one or more fleet model codes (chips). MB51 is filtered to transactions whose order matches an IW39 row whose sortField matches a Fleet Master unit on a selected model. Each fleet model is its own bucket. Materials qualifying on 2+ models route to a MULTI bucket (§7i). Preview line shows models / units / qualifying historical work orders.

Manual

Paste materials or work orders (see §3). Default for User-list assessments; available everywhere. Single bucket.

By Classification

Inventory-Master-driven: Inventory Type ∈ {…} AND MRP classifier ∈ {…} AND net movement in [min, max]. All filters AND together. Single bucket.

By Vendor

Multi-select of vendors from the Material→Vendor mapping (the drop zone appears once this tab is active). One bucket per vendor; materials under 2+ vendors go to MULTI.

07Analysis methodology ★

This is the section that matters when an output doesn't match expectations. Every number on the Analysis page and in the deliverables comes from one of the formulas below. The engine is shared/pipeline.js (deterministic, no LLM).

7a · Net consumption

Net consumption rule IF movementType ∈ {261, 201} → +|quantity| (issue)
IF movementType ∈ {262, 202} → −|quantity| (return)
ELSE → row not counted toward rate (109 receipts feed only the SOH back-calc, §7c)
  • Quantity is taken absolute, then signed by movement type — correct whether or not SAP already signed it.
  • More returns than issues over the window → routes to GREY for manual review.
  • Zero qualifying transactions → no data; the material drops out of the qualifying set (net < threshold).

7b · P1 baseline window & P2 rolling window

WindowDivisorUsed for
P1Fixed: p1Start → p1End (factory 2025-04-01 → 2025-08-31)5 monthsBaseline rate, compared to P2 (rate-change %)
P2Rolling: (runDate − p2Months) → runDate (factory p2Months = 3)p2MonthsThe Min/Max recommendation driver
Period rate rate = (Σ issues − Σ returns) ÷ months · flag = OK | NO_DATA | NEGATIVE_NET

P2 stockout anchoring (APP-E11). Before computing the P2 rate, the engine checks the back-calc tail (§7c). If the most recent period is an ongoing stockout > 7 days, the P2 window re-anchors at the last-consumption date instead of the run date — otherwise the trailing zero-consumption stockout days would masquerade as "demand dropped." The chosen anchor is reported per material (runDate vs lastConsumption).

7c · Stock-on-hand back-calc & stockout-aware drop detection ★ (APP-E1)

A sharp drop in recent consumption has two very different causes: demand genuinely fell, or stock ran out so nothing could be issued. They demand opposite actions. To tell them apart the engine reconstructs the site stock-on-hand history (shared/inventory-back-calc.js).

SOH back-calc Anchor on the last-consumption date; window = that date back socBackCalcMonths (factory 6), then forward to today.
Walk MB51 day-by-day from the current totQtyOh snapshot, applying signed movements:
  receipts (109) add stock · issues (261/201) remove · returns (262/202) add back.
Day-keys are computed in UTC (APP-FIX-BACKCALC-TZ) so they line up with the raw posting-date strings.
  • Stockout windows — contiguous spans where reconstructed SOH ≤ 0. Each carries start / end / day-count.
  • Rate-drop cause — when a sharp drop is detected, the engine classifies it as STOCKOUT_DRIVEN (a stockout window overlaps the P2 window) or GENUINE_DEMAND_DROP (stock was available; consumption fell anyway).
  • Stockout-dominated — if more than one stockout window falls inside the chosen P2 window, P2 is too unreliable for a verdict; the traffic light is forced GREY (decision-tree rule 0, §7h) and the rate-change flags are suppressed.
The SOH line is reconstructed, not pulled from SAP The stock-on-hand line and stockout bands are back-calculated from the current snapshot + MB51 movements (site stock only; 3PL receipts excluded). They are an analytical estimate. If MB51 was exported without 109 receipts, the line shows drawdowns only — re-export with 109 included. A negative reconstructed SOH is a signal that an assumption is wrong (missing movement types, stale snapshot, timing mismatch) — investigate the cause; it is never silently clamped.

7d · Inventory Adjustment detection

Cycle counts / stock adjustments post as issue transactions (often 201) and hit many materials on the same date, making rates look spiky and patterns look LUMPY. The detector finds these dates statistically and lets you confirm them.

Inv-Adj day detection For each posting date, count issue transactions (261 + 201). Compute mean μ and stddev σ across daily counts.
A date with count ≥ μ + N·σ is a candidate. Default N = invAdjSigmaThreshold = 5.

A modal lists each candidate (day-of-week, count, σ-above-mean, materials affected, total qty) and you tick the ones to exclude from rate calculations. Confirmed dates persist on parameters.invAdjConfirmedDates. P1, P2 and Adjusted-P2 rates all skip excluded dates; the chart shows them as dashed purple vertical lines, and the detail panel + Excel + PDF carry a distinct Inv-Adj table. Re-open the modal anytime via the "Inv Adj review" button.

7e · High Consumption Event (HCE) detection

HCE rules — period-scoped A) WO qty ≥ hcePctThreshold × period total (default 50%)
B) WO qty ≥ hceMultThreshold × average WO qty in period (default 3×)

When an HCE is found in P2, an Adjusted P2 Rate is also computed with the HCE work order(s) excluded — "what would the rate be without that one big spike." The official P2 Rate is unchanged; both are reported. HCE runs for P1 and P2; the Excel deliverable surfaces P2 HCEs (they affect Min/Max).

7f · Lumpy vs Smooth classification

Lumpy rules — full-window (A) top-WO share ≥ lumpyTopWoThreshold (default 40%)
(B) coefficient of variation > lumpyCvThreshold (default 1.2)
(C) WO count ≤ 3 AND total qty ≥ threshold

Any of the three → LUMPY, else SMOOTH. LUMPY materials are poor fits for a fixed Min/Max, so the engine recommends V1 (consumption-driven) for them (see §7g).

7g · Recommended Min / Max + MRP-type reclass ★ (APP-E8)

monthsBased method (default) recMin = round(P2 rate × minMonths)   (default 3)
recMax = round(P2 rate × maxMonths)   (default 6)

If P2 rate is 0 / NO_DATA / NEGATIVE_NET, both are null and the material lands in GREY. The leadTimeBased method (rate × leadTimeMonths + safetyStock) is available when Lead Times data is present.

APP-E8 — Min/Max only works under MRP type V1 Min/Max is computed for every material, including those whose MRP type is PD — especially PD items that are misclassified in SAP, since those are exactly what we need to surface. But Min/Max settings only operate under V1; PD cannot legitimately hold Min/Max. So whenever the engine recommends a Min/Max for a PD material, it also recommends reclassifying the MRP type to V1 — never dropping the recommendation to hide the issue.

Concretely: the Recommended MRP Type resolves to V1 when a Min/Max is recommended on a current-PD material (and also for LUMPY / HCE patterns). When that happens the material carries a reclass flag + a plain-English note ("PD cannot hold Min/Max — reclassify MRP type to V1 to apply the recommended Min/Max"). This surfaces as the Reclass column on the Analysis list (filterable — isolate every PD → V1 candidate in one click), a highlighted row + note in the detail panel's MRP comparison, a note in the Excel per-material sheet + PDF page, and a token in the LLM prompt context. The LLM only annotates it — the deterministic math decides.

7h · Traffic-light decision tree (priority order)

The assess() function evaluates rules top-down; the first match wins. A few-events overlay may then upgrade GREEN/BLUE to ORANGE.

#ConditionResultAction shown
0Stockout-dominated — >1 stockout window inside the P2 window (§7c)GREYStockout-dominated recent-rate window — verify supply continuity before changing Min/Max
1P2 flag ≠ OK OR P2 rate = 0GREYRefer for manual review — no recent (P2) consumption
2MRP type = NOT IN MASTERGREYNot in Inventory Master — excluded
3Recommended Min not calculableGREYRecommended Min/Max not calculable
4MRP ∈ wrMrpTypes (default PD) AND runway > wrHardMonths (9)PURPLELikely Working Redundant — review for destocking / write-down
5MRP ∈ wrMrpTypes AND runway > wrSoftMonths (6)PURPLEPossible Working Redundant — review stocking necessity
6PD AND total ≥ threshold (runway ≤ wrSoft)REDChange MRP type to V1. Set Min/Max to {recMin}/{recMax}
7V1 AND rec matches current Min & MaxGREENNo action required
8V1 AND rec pulls Min/Max downBLUELower Min/Max — low-risk reduction
9V1 AND rec pulls Min/Max upORANGERaise Min/Max — current insufficient
10Anything elseGREYNot assessed — manual review

Few-events overlay: after the primary code, if unique issuing work orders ≤ 2 AND the code is GREEN/BLUE, it upgrades to ORANGE ("Only N event(s) — pattern unreliable"). A rate from one or two events reflects luck, not pattern. RED/PURPLE/GREY are not overlaid. The PD → V1 reclass flag (§7g) rides on top of whatever rule fires when a Min/Max is recommended on a PD item — RED already says it in words; PURPLE-PD items also carry the structured flag/note while keeping their "redundant" action.

7i · Buckets, MULTI detection & the analysis loop

A "bucket" is the analysis unit; the full assessment runs once per material per bucket.

  • Fleet — one bucket per model + a MULTI bucket for materials qualifying on 2+ models (removed from per-model buckets to avoid double-counting).
  • Manual / User list / By Classification — single bucket.
  • By Vendor — one bucket per vendor + MULTI for materials under 2+ vendors.
  • Parameter Search — single bucket = the resolved filter intersection.

Multi-model qualification: net consumption ≥ threshold in each of 2+ models; the MULTI bucket unions that material's transactions across qualifying models.

08Charts — what they show

The material chart is an inline SVG. X = time (the analysis window); the main Y = cumulative net consumption. A chart toolbar lets you toggle the Consumption and Stock-on-Hand series independently.

Series & markers

  • Amber step line — cumulative net consumption. Steps up on issue dates, down on returns.
  • Violet line — back-calculated stock on hand over the window (APP-E1; see §7c). Toggleable.
  • Red wash bands — stockout windows (reconstructed SOH ≤ 0).
  • Orange dashed marker — the last-consumption date (the anchor for the back-calc + P2 re-anchoring).
  • Cyan / green dashed chords — P1 / P2 trend lines; slope = the period rate, drawn between actual cumulative values at the period endpoints (filled circles).
  • Background tint — cyan strip = P1 window; green strip = P2 window.
  • WO annotations — top HCE events get a marker + leader + label (WO / description / equipment / qty).
  • Purple dashed verticals — confirmed Inventory-Adjustment dates (§7d).
  • LUMPY badge — top-left when the material is LUMPY.

The chart caveats explicitly that the SOH line is back-calculated, not pulled from SAP. If a material has only a single posting date, the cumulative series is padded with a synthetic zero-point so the step line still draws; the empty-state ("no consumption transactions found") only triggers on a true data-absence case.

11LLM review surface

Provider-agnostic (Anthropic / OpenAI), configured on Settings (§12.2). "Run LLM review" on a detail panel captures the chart as PNG and sends it with a structured prompt (the material's stats + current recommendation), parsing a JSON response: verdict (ok / tweak / review), signals (from a fixed list), notes, and optional suggestedEdits. Suggested edits are shown, never auto-applied. The LLM annotates; the math decides.

11a · Operational Context

Every prompt begins with a generic Operational Context block ({customerContext}) so the model knows it's reviewing, e.g., an open-pit mining client — without any client identifiers crossing the boundary. Pick a fixed-pick library entry (Open-pit mining / Underground / Heavy civil / Minimal) or author a Custom slot (capped 300 chars, privacy-linted for IDs, company suffixes, all-caps codes, emails, IPs). Selection lives in settings.client.* — never on the canonical JSON.

11b · Signal-aware prompt

Tokens available to the template include {customerContext}, {netSign}, {rateChangeFlag} (incl. the stockout-driven vs genuine-demand-drop cause from APP-E1), {stockoutSummary}, {lastConsumptionDate}, {rateDropCause}, {recMrpType} + the APP-E8 reclass note, plus the per-material stats. A "WATCH FOR" block asks the model to name the specific signal that drove its verdict.

11c · Mass LLM Review

Select up to 50 materials and run the LLM against each in sequence, producing a consolidated results table with cancel / pause / resume. Verdicts use a distinct palette (cyan ok / amber tweak / magenta review) so they're never confused with the algorithmic traffic light. Drill into any row to select that material on the main page.

In-memory only — never persisted Mass-review (and single-review) LLM results live only in JavaScript memory. Nothing is written to localStorage/IndexedDB. Tab close / reload wipes everything. Download the Excel / JSON to keep it. The mass-review JSON is deliberately separate from the canonical intake JSON — math stays clean and rollback-safe; LLM annotations layer on top and can be re-loaded later via "⤒ Load mass review".

13Calibre Trace ★

Calibre Trace reconstructs the procurement chain — from purchase-requisition creation through PO, goods receipt, and first consumption — so you can see real lead times and where chains stall or get cancelled. It lives as a sibling page in this app shell (the Trace nav link) and reads the same canonical intake (intake.current). It needs PR History (§4) and MB51; Inventory Master is used for the multi-material scope filters.

Scope

  • Single material (live). The left-rail material picker is built from PR History, ranked by PR count, searchable by material or description. Pick one to load its chains.
  • Multi-material (scope UI live; cross-material views queued). Use Tune's list (User-list assessments), paste a list, filter by material group, or filter by stock level (out of stock / below Min (V1) / below Safety (PD) / above Max (V1)).

Scope + active view persist in trace.viewState across reloads.

Views (single-material — all live)

  • Procurement Chain (default) — a swimlane where each row is one PR/PO chain, broken into five phases (A–E) by elapsed days, plus a funnel cascade (PR generated → cancelled → PR-only → progressed to PO → delivered) with counts and units. Hover a bar for all milestone dates, the phase breakdown, total lead time, qty, and source (manual PR vs MRP-generated).
  • Phase Distribution — five box plots (Q1 / median / Q3 / whiskers capped at the Tukey fence / outliers) one per phase, a total-lead-time chevron, and a stats table. Needs ≥ 2 complete chains.
  • Volume cumulative — cumulative units over time: PR Raised, PO Raised, Site Receipt (109), Consumed (261), plus cancelled-PR ticks (vertical red lines) and a KPI strip (cancelled count / units / rate, delivered units, and a per-material Pack size multiplier).
  • Raw Data — every chain in a table (PR / dates / PO / 3PL GR / Site WH / first 261 / phases A–E / total / qty / state), with a per-row exclude/include toggle.

Chain state machine

Operator-locked rule: if there is a PO, ignore the PR's final state — a PO means the need was approved; post-PO cancellation is admin-meaning only. A PR with no PO is treated as cancelled (denied). States, in precedence order:

StateMeaning
CANCELLEDPR, no PO, deletion flag — red-tinted; kept out of the swimlane visual + funnel but visible in Raw Data.
PR_ONLYPR, no PO, not cancelled — open demand not yet ordered.
IN_FLIGHTPO but no site receipt (109) yet.
NOT_YET_CONSUMEDReceived but no consumption (261) yet.
COMPLETEPO + receipt + consumption — the only state counted toward average total lead time.

Phase colours on the swimlane: A  B  C  D  E. An admin-cancel (PO present but PR deletion flagged) renders in its actual phase colour with an advisory annotation, not red.

Filters & interactions

  • Year filter — All years + one button per year in the data.
  • Sigma trim — Off / Loose 3σ / Standard 2σ / Tight 1.5σ — drops chains slower than mean + Nσ of total lead time.
  • Manual exclude/include — per-row toggle in Raw Data, or right-click a swimlane bar ("Exclude this chain" / "Show in Raw Data view…"). Excluded chains stay visible on the swimlane ghost-dulled (~10% opacity) rather than disappearing (APP-FIX-SWIM-GHOST). "Reset manual" clears them. Excludes persist per material.

Diagnostics — why complete chains are missing

If a material has chains but none reach COMPLETE, a panel diagnoses the cause: no 109 receipts in MB51 (often a consumption-only export filter — re-export with 109); receipts present but blank PO field (assessment parsed before the MB51 purchaseOrder alias — re-drop the MB51 file and re-parse); receipts whose POs aren't in PR History (PR-History window too short); or PO numbers that match but a date-ordering edge case prevented classification.

Queued (multi-material) views

Built one at a time as operator priority dictates: Internal Process MoM, Supplier Performance (scorecard + OTIF), 3PL Performance, Cancellation Diagnostic, Unit-Cost Sensitivity, Year-on-Year, and a multi-material Data Table. The single-material Year-on-Year port (PORT-6) is also queued.

14Troubleshooting

Chart shows "no consumption transactions found" The material has zero qualifying MB51 rows (no 261/201/262/202). Causes: wrong movement-type filter on the export; material-number case mismatch between MB51 and Inventory Master; or the file is all transfers/receipts.
The stock-on-hand (violet) line looks wrong / shows only drawdowns The SOH line is back-calculated and needs 109 receipts in MB51 (§7c). If 109 is absent, re-export MB51 with receipts included. A reconstructed SOH that goes negative is a signal that a data assumption is off (missing movement types, stale snapshot, timing) — investigate; it is never silently clamped.
A PD material shows "PD → V1" in the Reclass column Expected (APP-E8). The engine recommends a Min/Max for it, and Min/Max only works under MRP type V1 — so it's surfacing a misclassified PD item that should be V1. Filter the Reclass column to find them all.
Recommendations differ from a prior run / the legacy Python output Check: (1) parameters match (compare the JSON parameters block); (2) the threshold is identical (default 10); (3) the MB51 ↔ Inventory Master reconciliation % is the same — different join coverage changes the qualifying set; (4) whether stockout-aware re-anchoring (§7b/7c) changed a stockout-affected material's verdict — that is intended.
Trace shows no chains / "Need PR History" Trace needs PR History uploaded on Intake and a re-save. If chains exist but none are COMPLETE, read the on-page diagnostic panel (§13) — it names the cause.
LLM review fails with "Failed to fetch" Usually CORS from opening via file://. Serve over HTTP: from the app folder run python -m http.server 8000 and open http://localhost:8000.
Start clean Settings → About → Wipe all local data clears every saved intake, alias override, and LLM key (or DevTools → Application → Clear site data).
