01Overview

The Inventory Optimization App replaces the legacy Python-script + Excel-handoff workflow with a self-contained, browser-only tool. Drop SAP exports on the intake page, the app validates them, lets you pick a scope, runs the consumption-profiling pipeline, lets you review per-material findings (with optional LLM second opinion), and ships an Excel pack matching the legacy deliverable shape.

The app is deterministic. Same inputs + same parameters always produce the same canonical JSON, the same recommendations, and the same Excel output. The LLM, when invoked, only annotates — it never changes the numbers.

Architecture:

  • Intake engine (intake/intake.html) — upload, schema-map, DQ-gate, scope, parameter, review, export.
  • Canonical JSON (shared/canonical-schema.js) — the contract between intake and analysis. Versioned (currently 1.0.0).
  • Analysis engine (analysis/analysis.html) — consumes the JSON, runs the pipeline, surfaces per-material detail, hosts the LLM review, exports Excel.
  • Settings (settings/settings.html) — parameter defaults, LLM keys, alias overrides, factory reset.

02Quick start

  1. From the Dashboard, click Intake.
  2. Assessment name (mandatory ★) — type a name in the header. Saves and "Open in Analysis" both refuse to proceed without one.
  3. Batch mode (optional, v2.1) — if you have a previously-saved intake with the heavy data already loaded, click Reuse common data → modal lets you pick which datasets (MB51 / IW39 / Fleet / Inv Master) to hydrate. Skip this section if you're starting fresh.
  4. Step 0 — Assessment type. Pick one of UNIT/FLOC, User list, or Parameter Search. Non-applicable upload zones grey out.
  5. Step 1 — Upload. Drag SAP exports onto the highlighted drop zones. Each card shows row counts and cross-file match percentages.
  6. Step 2 — Schema mapping. Confirm auto-mapped headers; manually override any unmatched columns.
  7. Step 3 — Data Quality gate. Green = continue, amber = warnings shown but continue, red = halt and fix.
  8. Step 4 — Scope. For UNIT/FLOC, pick fleet model(s) / classification / vendor. For User list, paste materials directly OR upload an optional file. For Parameter Search, configure filters in Step 4b.
  9. Step 5 — Parameters. Per-run override of saved defaults. Reset button reverts overrides without touching Settings.
  10. Step 6 — Final scope summary. Confirms exactly how many materials will be analyzed, bucket-by-bucket. Mismatch advisories shown.
  11. Step 7 — Review & Export. Save to localStorage, download .json, or "Open in Analysis Engine →" to run the pipeline.
  12. On the Analysis page: pick a bucket, filter / sort the material list, click a row for the detail panel. Right-click a row to mark for LLM review or PDF print. Use Export this bucket for the per-bucket workbook, Export ALL (combined) for the master index, Export PDF Pack for client deliverable, Mass LLM Review for batched chart annotations.

Batch-mode workflow (v2.1)

Running multiple analyses against the same SAP extracts (e.g. different fleet models or material lists)? Don't re-upload — use batch mode:

  1. Run + save your first intake normally.
  2. Return to the Intake page. A new "Batch mode" tile appears below the Intake Engine header.
  3. Click ↻ Pick source + datasets… → modal opens.
  4. Pick the source intake and tick the datasets to hydrate. Defaults: heavy common files checked (MB51, IW39, Fleet, Inv Master), specifier files unchecked (userList, materialVendor, leadTimes). Adjust as needed.
  5. Click ↻ Load selected datasets. The drop cards show ↻ reused from "…" instead of "drag here".
  6. Assessment type, scope and parameters are NOT inherited — pick fresh in Step 0 onwards.
  7. Save under a new name (auto-suggested: {source} — batch HHMM).

04Data files & SAP exports

MB51 — Material Movements

The transactional backbone — every goods issue and return.

Required columns (alias-resolved, case-insensitive, punctuation-stripped):

  • postingDate — Posting Date / Pstng Date / Date
  • order — Order / Order Number (joins to IW39)
  • material — Material / Material Number
  • description — Material Description / Material descr.
  • quantity — Quantity / Qty (always taken as absolute value; sign comes from movement type)
  • movementType — Movement Type / MvT

Movement types the pipeline recognises:

CodeMeaningSignIncluded in
261Goods issue to order+ (consumption)P1, P2, cumulative
201Goods issue to cost center+ (consumption)P1, P2, cumulative
262Reversal of 261 (return)− (return)P1, P2, cumulative
202Reversal of 201 (return)− (return)P1, P2, cumulative
All other movement types are dropped silently during processing (transfers, receipts, etc.).

The DQ gate warns if the file spans < 12 months, if 261 is missing, or if 262 is missing (in which case net = gross).

IW39 — Work Orders

The bridge from MB51 transactions to fleet equipment. Required only for UNIT/FLOC assessment type.

  • order — joins to MB51
  • sortField — equipment unit (e.g. TD403); joins to Fleet Master
  • basicStartDate — used to filter out future-dated orders
  • description — WO description, surfaces in HCE annotations on charts

Fleet Master

Equipment-unit roster. Required for UNIT/FLOC.

  • model — fleet model code (e.g. 775G, D9T); defines buckets
  • sortField — equipment unit, bridges through to IW39
  • unitType, manufacturer, functLocDescription — display only

Inventory Master

Stock + MRP settings + classification metadata. Required for all three assessment types.

  • material — join key against MB51 materials
  • totQtyOh — current stock on hand (drives Runway @ P2)
  • totValueOh — current on-hand value (optional, used in Parameter Search)
  • mrpInd — MRP type (PD, V1, ZE…) — feeds the traffic-light rules
  • mrpMin / mrpMax — current MRP min/max
  • inventoryType — NORM / INSP / etc. — filter dimension
  • materialGroup / manufacturer — Parameter Search dimensions; surfaced on per-material Excel sheet
  • primaryVendor — optional vendor fallback

Missing optional columns degrade gracefully — that filter dimension just shows "no values available".

Material → Vendor mapping

Conditional. Required only when scope mode is "By Vendor".

Lead Times (deferred)

Per-material lead-time + safety-stock data. Optional. Enables leadTimeBased Min/Max method when present. The data isn't typically available yet; v1 schema accommodates it for the future without breaking changes.

05Scope selection

For UNIT/FLOC assessments, the scope tab drives which materials get bucketed and how.

Fleet

Pick one or more fleet model codes. MB51 is filtered to transactions whose order matches an IW39 row whose sortField matches a Fleet Master unit attached to a selected model. Each fleet model becomes its own bucket. Materials qualifying on 2+ models are routed to a MULTI cross-fleet bucket (see §7g).

By Classification

Inventory-Master-driven filter: Inventory Type ∈ {…} AND MRP classifier ∈ {…} AND movement amount in [min, max]. All filters AND together. Single bucket.

By Vendor

Multi-select of vendors from the Material→Vendor mapping. One bucket per vendor. Materials present under 2+ vendors go to MULTI.

06Parameter Search

Only visible when assessment type = Parameter Search. PBI-style filter builder.

Dimensions

  • MB51 movement — net consumption over the analysis window (numeric)
  • Material Group — from Inventory Master (string)
  • Manufacturer — from Inventory Master (string)
  • Value on hand — totValueOh from Inventory Master (numeric)
  • Qty on hand — totQtyOh from Inventory Master (numeric)
  • Inventory Type — from Inventory Master (string)
  • MRP indicator — from Inventory Master (string)

Filter modes

Each filter card has a Simple mode and an Advanced mode:

  • Simple, string: chip multi-select of distinct values
  • Simple, number: min / max numeric inputs
  • Advanced, string: comma-separated value list, or regex (regex overrides values when set)
  • Advanced, number: same min / max with precise decimals

Multiple filters AND together — a material must satisfy every active filter to qualify. Live preview shows matching material count, total net consumption, and total on-hand value.

07Analysis methodology ★

This is the section that matters when an output doesn't match expectations. Every number on the analysis page and in the Excel deliverable comes from one of the formulas below. Source: ports of scripts/02_extract_model.py, 03_build_charts.py, 04_mrp_analysis.py — semantics preserved line-for-line.

7a · Net consumption

The atomic unit of measurement. For every MB51 row:

Net consumption rule IF movementType ∈ {261, 201} → +|quantity| (issue)
IF movementType ∈ {262, 202} → −|quantity| (return)
ELSE → row dropped

Aggregated per material, this gives the Total Consumed column in the Index sheet and the chart's Y-axis (cumulative).

Notes:

  • Quantity is taken as absolute value, then signed by movement type. If SAP exports quantities already signed, this is still correct because |q| reverses sign back to magnitude.
  • Negative net consumption (more returns than issues over the period) → flagged as NEGATIVE_NET; rate forced to 0 and material routes to GREY for manual review.
  • Materials with zero qualifying transactions get NO_DATA.

7b · P1 baseline window & P2 rolling window

Two windows are computed per material:

Window definitionMonths divisorUsed for
P1Fixed: p1Start → p1End from parameters (default 2025-04-01 → 2025-08-31)5 (hard-coded)Baseline rate to compare against P2 (rate-change %).
P2Rolling: (runDate − p2Months) → runDate (default 3 months back from run date)parameters.p2MonthsThe Min/Max recommendation driver. P2 reflects current consumption.
Period rate formula rate = (Σ issues − Σ returns) ÷ months
rate flag = OK | NO_DATA | NEGATIVE_NET

If P1 rate is 0 and P2 rate is positive, this is a new draw — the material started consuming inside the recent window. Rate Change shows as N/A because there's no P1 baseline to divide by.

7b.5 · Inventory Adjustment detection ★ v2.0

SAP cycle counts and stock-adjustment events post as issue transactions (typically movement type 201, goods issue to cost center). They capture accumulated bookkeeping rather than genuine consumption, and because they hit many materials on the same date, they make the rate look spiky and the pattern look LUMPY when the underlying consumption is in fact smooth.

The detector finds these dates statistically:

Inv-Adj day detection For each posting date in MB51, count issue transactions (movement types 261 + 201).
Compute mean μ and stddev σ across the daily-count distribution.
A date with count ≥ μ + N·σ is a candidate.
Default N = invAdjSigmaThreshold = 5. Lower (2–3σ) → more candidates; raise (7–10σ) → only extreme spikes.

The operator confirms. After the pipeline runs, a modal pops up (only when candidates exist and the canonical JSON has no prior confirmations) listing each candidate date with: day-of-week, transaction count, sigma-above-mean, materials affected, and total quantity. The operator ticks the dates to exclude from rate calculations. Skipping leaves rates unchanged; confirming re-runs the pipeline with those dates excluded.

Confirmed dates persist on parameters.invAdjConfirmedDates in the canonical JSON. Reloading the same intake remembers the confirmations.

How Inv Adj exclusion works

Same mechanic as HCE but labelled differently:

  • P1 and P2 rate calculations skip any transactions whose posting date is in invAdjConfirmedDates.
  • The Adjusted P2 rate (already used for HCE exclusion) also excludes Inv Adj dates — both signals stack.
  • Per-material results carry an invAdj array, rendered as a distinct table on the detail panel (purple-tinted) and a separate section in the Excel per-material sheet.
  • The chart shows vertical dashed purple lines at Inv Adj dates with an "INV ADJ" label — visually distinct from amber HCE work-order markers.

The "Inv Adj review" button on the analysis page re-opens the modal anytime — useful for tuning the sigma threshold or revisiting decisions.

7c · High Consumption Event (HCE) detection

A single work order is flagged as an HCE inside a period if either:

HCE rules — period-scoped A) WO qty ≥ hcePctThreshold × (period total)
     default: 50% of the period

B) WO qty ≥ hceMultThreshold × (avg WO qty in period)
     default: 3 × average

When an HCE is detected in P2, an Adjusted P2 Rate is also computed by recomputing the period rate with the HCE work order(s) excluded. This gives a "but what would the rate be without that one big spike" answer. The official P2 Rate is unchanged — both are reported.

HCE detection runs separately for P1 and P2. The reference Excel deliverable shows P2 HCEs only (because they affect the Min/Max recommendation), but P1 HCEs are preserved in the per-material JSON output for completeness.

7d · Lumpy vs Smooth classification

Operates on all qualifying transactions in the analysis window (not period-scoped). A material is LUMPY if any of:

Lumpy rules — full-window scope (A) top-WO share ≥ lumpyTopWoThreshold (default 40%)
(B) coefficient of variation (CV) > lumpyCvThreshold (default 1.2)
(C) WO count ≤ 3 AND total qty ≥ threshold (catches small-n big-batch materials that CV under-rates)

Otherwise the material is SMOOTH. The classification drives the Recommended MRP Type (see 7e) — Lumpy materials are recommended for V1 (consumption-driven) over PD (fixed Min/Max).

7e · Recommended Min / Max

Two methods are available; monthsBased is the v1 default. leadTimeBased is roadmap, requires per-material lead-time data not yet typically available.

monthsBased method (default) recMin = round(P2 rate × minMonths)    // default minMonths = 3
recMax = round(P2 rate × maxMonths)    // default maxMonths = 6
leadTimeBased method (roadmap) recMin = round(P2 rate × leadTimeMonths + safetyStock)
recMax = round(recMin + P2 rate × orderCycleMonths)

If P2 rate is 0 / NO_DATA / NEGATIVE_NET, both values are null and the material lands in GREY.

The Recommended MRP Type on the per-material Excel sheet is derived: V1 when the pattern is LUMPY or any HCE was flagged in P2 (a fixed Min/Max would be a poor fit), otherwise mirrors the current MRP type. This is a heuristic — the human reviewer can override.

7f · Traffic-light decision tree (10 rules + few-events overlay, priority order)

The assess() function evaluates rules in order; the first matching rule wins. After the primary code is set, a few-events overlay may upgrade GREEN/BLUE to ORANGE if the material's consumption was statistically thin.

#ConditionResultAction shown
1P2 flag ≠ OK OR P2 rate = 0GREYRefer for manual review — no recent (P2) consumption
2MRP type = NOT IN MASTERGREYMaterial missing from Inventory Master — exclude or fix master first
3Recommended Min not calculableGREYRecommended Min/Max not calculable
4MRP type ∈ wrMrpTypes (default PD) AND stock-runway > wrHardMonths (default 9 mo) at P2 ratePURPLELikely Working Redundant (WR) — review for destocking / write-down
5MRP type ∈ wrMrpTypes AND stock-runway > wrSoftMonths (default 6 mo) at P2 ratePURPLEPossible Working Redundant — review stocking necessity
6PD AND total ≥ threshold (and stock-runway ≤ wrSoftMonths)REDChange MRP type to V1. Set Min/Max to {recMin}/{recMax}
7V1 AND rec matches current Min & MaxGREENNo action required
8V1 AND recommended pulls Min/Max down from currentBLUELower Min/Max — low-risk reduction
9V1 AND recommended pulls Min/Max up (insufficient)ORANGERaise Min/Max — current setting insufficient
10Anything else (other MRP type, missing data)GREYNot assessed — manual review

Few-events overlay (priority bump)

After the primary code is determined, the pipeline counts the number of unique issuing work orders in the analysis window. If woCount ≤ 2 AND the primary code was GREEN or BLUE, the outcome is upgraded to ORANGE with an explicit "few events" rationale. The rate computed from one or two events isn't statistically reliable — a single rebuild WO at the right moment can produce a rate that looks fine but reflects luck, not pattern. RED, PURPLE, and GREY are not overlaid (they're already review-priority).

Few-events rule IF woCount ≤ 2 AND primary code ∈ {GREEN, BLUE}
    → upgrade to ORANGE · action: "Only N event(s) — pattern unreliable, manual review"
ELSE → primary code stands

About the PURPLE / Working Redundant rule

A material on a watched MRP type (default PD — fixed Min/Max, MRP-driven replenishment) but with accumulated stock above a configurable runway threshold flags as "the Min/Max is probably too high, OR this item shouldn't be stocked at all". The threshold escalates:

  • wrSoftMonths–wrHardMonths runway on a watched MRP type → PURPLE · Possible WR. Review stocking necessity.
  • > wrHardMonths runway on a watched MRP type → PURPLE · Likely WR. Review for destocking / write-down.

Thresholds and watched MRP types are configurable in Settings → Parameter defaults and per-intake in Step 5. Factory defaults are wrSoftMonths = 6, wrHardMonths = 9 (lowered from 12 in v2.0.1 per operator feedback that 12 was too lenient), and wrMrpTypes = ['PD']. The action-text shown to the operator interpolates the active threshold and the specific MRP type that tripped — e.g. "Likely Working Redundant — 11.3 months of stock at P2 rate (>9mo with PD). Review for destocking / write-down."

The check applies only to materials with non-zero stock and a positive P2 rate. V1 materials are not in the default watch-list — V1 is consumption-driven, so the system will naturally stop replenishing as stock draws down.

The traffic-light value is what colours the cell in Excel + on the analysis page. The Action Required text is rendered verbatim from the rule.

7g · Buckets, MULTI detection, and the analysis loop

"Bucket" is the analysis unit. The pipeline runs the full assessment (period rates → HCE → lumpy → traffic light) once per material per bucket.

Bucketing strategy:

  • Fleet — one bucket per fleet model. Plus a MULTI bucket containing materials that qualify on 2+ models. MULTI materials are removed from the per-model buckets to avoid double-counting.
  • Manual / User list — single bucket.
  • By Classification — single bucket.
  • By Vendor — one bucket per vendor. Plus a MULTI bucket for materials present under 2+ vendors. Same de-duplication rule.
  • Parameter Search — single bucket whose materials list is exactly the resolved filter intersection.

Multi-model qualification rule: a material's net consumption ≥ threshold in each of 2+ models. The MULTI bucket's transactions are the union of that material's transactions across all qualifying models.

08Charts — what they show and why some look the way they do

The cumulative consumption chart

X axis: time, spanning the full analysis window (typically the MB51 export range). Y axis: cumulative net consumption (units), starting at 0 on the left.

  • Amber step line — actual cumulative net consumption over time. Steps up on issue dates, down on return dates. Built from MB51, grouped by unique posting date.
  • Cyan dashed chord — P1 trend. Connects (P1 start, cumulative at P1 start) → (P1 end, cumulative at P1 end). The slope equals the P1 rate; the label shows the rate.
  • Green dashed chord — P2 trend. Same construction over the P2 window. Labelled with P2 rate.
  • Filled circles at the start and end of each trend chord — these mark the actual cumulative values at the period endpoints. The chord is literally drawn between two points on the cumulative curve, so the slope is the average net consumption rate over that period.
  • Background tinting — cyan strip = P1 window; green strip = P2 window.
  • WO annotations — top-3 HCE events by quantity get a marker dot + leader line + label (WO number, description, equipment, qty).
  • LUMPY badge — top-left, only when the material was classified LUMPY.
  • Legend — top-right.

Why a chart might not render the expected shape

Known issue addressed in v1.1.0 Prior to v1.1.0, the chart bailed with the message "no consumption data" when the material's cumulative series collapsed to a single posting date — even though P1, P2 rates and recommendations all showed numbers. This was a real bug, not a data issue, and it's fixed.

The single-date collapse — why the bug existed

The cumulative series is built by grouping transactions by unique posting date. If a material had:

  • One work order with 12 line items all posted on the same date — that's one cumulative point.
  • Multiple WOs that happened to share the same posting date — still one cumulative point.

The chart's polyline needs at least two points to draw a step line. The rates (P1 / P2) bypass this problem entirely because they sum quantities directly without grouping by date.

The fix

Cumulative-series padding now applies automatically:

  • If the first cumulative point's date is after the P1 start AND its value isn't already 0, a synthetic {date: p1Start, cum: 0} point is prepended.
  • If the last cumulative point's date is before the P2 end, a trailing flat {date: p2End, cum: lastValue} point is appended so the chart shows the post-event plateau.
  • If, after both passes, only one point remains (genuinely a single date with no later flat tail), the synthetic 0-point is added regardless so the step line has somewhere to step from.

The empty-state message now reads "no consumption transactions found" and only triggers when zero qualifying transactions exist for the material — a true data-absence case, not a date-collapse case.

Other appearance quirks worth knowing

  • Trend chord "looks too low" — earlier versions anchored the trend at the cumulative value just before the period started; if there was a big jump on day 1 of the period, the trend visually started below the curve. Since v1.0.0, trends anchor on actual cumulative values at both endpoints — the chord is a straight line between two points on the curve, not a projection from the pre-period level.
  • P2 chord with no P1 data — if P1 flag ≠ OK, only the P2 chord renders. The chart still draws; the P1 background tint just sits empty.
  • Cumulative goes down — happens when a return (262/202) hits before the next issue. Y axis extends below 0 if cumulative ever turns negative.
  • Lone tall step at the right edge — typical for a material whose only consumption was a single recent rebuild WO. The chord still shows the rate; the badge will read LUMPY.

10b · PDF Pack ★ (v2.0)

Per-material PDF deliverable — one A4 portrait page per selected material, designed for client presentation rather than working analysis (Excel is the working format).

Marking materials

  • Right-click any row in the material list to open a context menu. Pick Mark for PDF print to add to the PDF queue, Mark for LLM review to add to the Mass-LLM queue. The same menu lets you Unmark or Open the detail panel.
  • Marked rows show small badges next to the material number (✦ review · ⤓ PDF) and a coloured stripe down the left edge of the row.

  5. Embedded chart — the same SVG-rendered cumulative chart with P1/P2 trend chords, HCE annotations, Inv Adj vertical lines, LUMPY badge. ~186 mm × 73 mm.

  7. MRP Settings Comparison — Current vs Recommended, with amber row highlight where they differ.
  8. HCE events table (only when material has HCE flags) — orange-tinted, lists each HCE work order with date, equipment, qty, % of P2.
  9. Inv Adj events table (only when material has confirmed Inv-Adj dates) — purple-tinted, lists each adjustment date with order, equipment, qty, reason.
  10. Footer — timestamp + version + repo URL on every page.

Implementation notes

  • Built on jsPDF 2.5.1 + jsPDF-autoTable 3.8.2, both CDN-loaded. No html2canvas dependency — the chart embeds via AppChart.toPng() (same path used by the Excel chart export), all text + tables are jsPDF native for crisp vector output and smaller file size.
  • Rendering speed: ~150–300 ms per material (chart capture is the bottleneck). 50 materials → ~15 s.
  • If the modal is closed mid-build, you get a "still running — close anyway?" confirm. PDF won't complete if you confirm close.

11LLM review surface

Provider-agnostic. Configure on the Settings page: pick provider (Anthropic / OpenAI), paste API key, pick model (model list is fetched dynamically from the provider).

On a material's detail panel, "Run LLM review" captures the rendered chart as PNG, sends it to the chosen provider's chat-completions endpoint with a multimodal input (image + structured prompt including the material's stats and current recommendation), and parses the JSON response:

{
  "verdict": "ok" | "tweak" | "review",
  "signals": ["negativeNet","sharpDrop", ...],
  "notes":   "specific analyst notes naming the signal seen",
  "suggestedEdits": [
    { "field": "recMin", "newValue": 15, "rationale": "…" }
  ]
}

Structured signals (v2.1.0): the model is asked to name the specific signal that drove its verdict from a fixed list — negativeNet, sharpDrop, sharpRise, spikeNoHce, flatTail, fewEvents, workingRedundant, seasonal, other. The signal-rendering chips on the detail panel land in v2.1.1; v2.1.0 captures the data and ships the prompt overhaul.

Suggested edits are not applied automatically — they appear as a list in the LLM panel; you decide what to action.

Browser CORS: Anthropic requires the anthropic-dangerous-direct-browser-access: true header (sent automatically). OpenAI accepts in-browser keys directly. Both expose the key to the page — fine for personal / internal use; route through a server proxy if ever shipped client-side.

If the call fails with "Failed to fetch", you likely opened the page via file:// — serve over HTTP instead: cd App/v2 && python -m http.server 8000.

11a · Operational Context (v2.1.0)

Every LLM prompt now begins with a generic Operational Context block injected as {customerContext}. The intent is to let the model know it's reviewing for, e.g. an open-pit mining client without putting any client identifiers across the browser → LLM-provider boundary.

Pick the active context in Settings → 03 · Operational context:

  • Open-pit mining (default) · Underground mining · Heavy civil / construction · Minimal — no context — fixed-pick library entries, deliberately generic. No fleet codes, no site names.
  • Custom (advanced) — operator-authored, capped at 300 characters, lint-checked for long numeric IDs, company suffixes (Inc / Ltd / etc.), 3+-character all-caps tokens (potential site codes), emails, and IPs. Lint warns but does not block — you confirm in a dialog before send.

Active selection and Custom text live in settings.client.* localStorage keys. Never written to the canonical intake JSON — the math deliverable stays free of LLM-coupling metadata.

11b · Signal-aware prompt (v2.1.0)

The factory prompt was restructured to surface the specific signals the operator cares about. New tokens available in the template editor:

  • {customerContext} — resolved Operational Context (see 11a)
  • {netSign} — POSITIVE / NEGATIVE (returns dominate) / MIXED / NO_DATA — surfaces materials flowing back to stores
  • {rateChangeFlag} — empty, or ⚠ SHARP DROP (P2 ≤ 60% of P1) or ⚠ SHARP RISE (P2 ≥ 160% of P1)
  • {invAdjCount} — count of operator-confirmed Inv-Adj dates excluded from rate
  • {daysSinceLastIssue} — integer days since last issue WO, or no issues in window

The new factory prompt also includes an explicit WATCH FOR block instructing the model to name specific signals (negative consumption, steep drop, spike-without-HCE, long flat tail, few events, working-redundant). Operators on the previous v2.0.x factory default are silently upgraded; customised prompts are preserved untouched — click ↺ Reset to factory in Settings if you want to adopt the v2.1.0 prompt.

11c · Mass LLM Review ★ (v2.0)

A second LLM entry point that lets you select up to 50 materials from the current bucket and run the LLM against each one in sequence, producing a single consolidated results table you can drill into and export.

Workflow

  1. On the analysis page, click ✦ Mass LLM Review in the Bulk operations row.
  2. Select view — modal shows the bucket's materials (TL · Material · Description · Total · P2/mo · Pattern). Tick checkboxes (max 50). "Select all visible" / "Clear" + counter "N / 50 selected". Click Review N materials.
  3. Run view — progress bar + live results table fills row-by-row as the loop calls the LLM once per material. Each row shows ⋯ pending → ⏳ in-flight → ✓ done (or ✗ on error). Cancel stops the loop without losing accumulated results; Pause halts after the current call and exposes a Resume action.
  4. Results view — final table, with three actions: ⤓ Download Excel, ⤓ Download JSON, ⌫ Close & Wipe.

The LLM verdict palette (distinct from the math TL)

Three values, intentionally a different palette so you don't confuse them with the algorithmic traffic light:

  • ok (cyan) — recommendation looks right given the chart
  • tweak (amber) — recommendation is close but a parameter change would improve it
  • review (magenta) — a planner needs to look at this manually (seasonal, anomaly, etc.)

Drill-down

Clicking any row in the results table closes the modal and selects that material on the main analysis page — chart + stats + LLM notes render in the detail panel using the cached LLM result from the batch. To return to the full table: click ✦ Mass LLM Review again.

Data security posture

In-memory only — never persisted The mass-review session lives only in JavaScript memory. Nothing is written to localStorage / IndexedDB at any point during the run. Single-review LLM results from this session are also held only in memory.

Page reload = data loss. The orchestrator picks "speed + clean security" over "resumability". You accepted this trade-off at design time.

v2.1.1: modal close no longer wipes. Closing the Results view now keeps all in-memory LLM data. The floating chip stays visible so you can re-open the Results view, and per-material verdicts remain available when you drill into a material on the main analysis page. Click ⌫ Clear LLM data in the Results view footer when you're done. Tab close / page reload still wipes everything (no localStorage).

v2.1.1: Excel-style filters on selection + results tables

Both the Mass LLM selection view (where you pick materials to send) and the results view (live + completed) have the same column-header sort + filter affordances as the main analysis list:

  • Set pickers (checkbox list) for traffic light / verdict / pattern / status
  • Numeric range pickers for totals / P2/mo / latency
  • Substring search for material number / description / notes
  • Click column label to sort; click ▾ caret to filter
  • "Select all visible" on the selection view respects the active filter (only adds filtered rows to selection)

Modal-local filter state — applying a filter in the modal does NOT change the main analysis list's filters.

Re-loading a saved review later

To revisit a prior batch:

  1. Open the canonical intake JSON for the assessment (analysis page boots from it).
  2. Click ⤒ Load mass review in the Bulk operations row.
  3. Select the saved Mass_LLM_Review_*.json file → modal opens in Results view with all your LLM annotations restored.
  4. Drill-down works as if the batch just ran.

The mass-review JSON is intentionally separate from the canonical intake JSON — math (intake JSON) stays clean and rollback-safe; LLM annotations (mass-review JSON) layer on top.

Background continuation

Closing the modal while the loop is running keeps the loop running. A floating chip "Mass review: N / M — Reopen" appears at the bottom-right of the page. Click Reopen to return to the Run view at any time.

The prompt template

Mass-review uses the same prompt template as single-review. Edit it in Settings → §03 LLM prompt template. Available placeholders: {material}, {description}, {bucket}, {pattern}, {totalNet}, {p1Rate}, {p2Rate}, {p2Months}, {rateChange}, {adjP2}, {hceText}, {woCount}, {mrpType}, {stock}, {cmin}, {cmax}, {runway}, {recMin}, {recMax}, {recMrpType}, {trafficLight}, {action}. Each mass-review JSON records the SHA-256 hash of the template that was active when the batch ran, for audit trail.

Excel deliverable

The Mass_LLM_Review_*.xlsx is the reference per-bucket workbook shape with three extra columns on the Index sheet:

  • LLM Verdict (column R) — coloured pill (cyan / amber / magenta), AutoFilter active.
  • LLM Notes (column S) — free-text analyst notes from the LLM.
  • LLM Latency (column T) — seconds per call, useful for cost diagnostics.

Each per-material sheet gets an additional "LLM Review" block at the bottom showing the verdict, notes, suggested edits, provider/model, prompt hash, and timestamp.

Error handling

  • Per-material API errors (HTTP 4xx/5xx, network drops, malformed JSON from the model) are captured in the row's status column. The loop continues with the next material.
  • Rate-limit (HTTP 429) → row marked error, loop continues. Re-run that material later by re-selecting it in a new batch.
  • If the LLM returns text that can't be parsed as JSON, the row's verdict defaults to review with notes "(could not parse model response — see raw)".
  • Cancel stops the next call from being issued. Materials whose calls have already returned stay in the results; uncalled ones are marked skipped.

12Troubleshooting

Chart shows "no consumption transactions found" The material has zero qualifying MB51 transactions (no rows of types 261/201/262/202). Common causes: ① wrong movement-type filter on the SAP export; ② case mismatch in material number between MB51 and Inventory Master; ③ all transactions in the file are non-issue types (e.g. transfers).
Numbers show in P1/P2 but chart looks sparse Fixed in v1.1.0 — was caused by single-date cumulative collapse. If still seeing this, hard-reload (Ctrl+Shift+R); GH Pages might be serving cached HTML.
Recommendations differ from prior Python output Check: (1) parameters match between the two runs (compare parameters block in the JSON against the Python script's defaults); (2) DQ gate's reconciliation % between MB51 and Inventory Master is the same — different join coverage will produce different qualifying material sets; (3) threshold setting is identical (default 10).
Export Excel fails with "sum.column is not a function" You're on a stale cached version of excel.js. Hard-reload, or in DevTools → Application → Storage → Clear site data, then reload. Fixed in v1.0.0 baseline.
LLM review fails with "Failed to fetch" Almost always a CORS issue from opening the page via file://. Serve over HTTP: from App/v1/ run python -m http.server 8000 then open http://localhost:8000.
"Wipe all local data" to start clean Settings page → About section → "Wipe all local data" clears every saved intake, every alias override, every LLM key. Can also be done via DevTools → Application → Clear site data.
Mass review modal won't open — "No LLM provider configured" Add a provider key in Settings → §02 LLM review providers, fetch + pick a model, then return. The Mass review button reads provider status at click-time.
Closed the mass-review modal before downloading By design, closing wipes all in-memory LLM data — there's no recovery in this session. Re-run the batch (LLM costs apply again). If you'd downloaded the mass-review JSON, you can re-load it via "⤒ Load mass review" once the canonical intake is open.
"Load mass review" rejects the JSON The bucket key in the saved JSON has to match a bucket in the current analysis. Make sure the matching canonical intake is loaded (the same assessment that produced the original batch). The error message names which bucket key was expected.
